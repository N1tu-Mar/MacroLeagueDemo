  ---
  1. The 30,000-foot view: two halves

  Almost every app is split into two halves that talk to each other:

  - Frontend ("client") = the app running on the user's phone. The buttons,
  screens, text. This is the src/ folder.
  - Backend ("server") = a computer somewhere else that stores everyone's data
  permanently and enforces the rules. This is the supabase/ folder.

  The phone doesn't own the data. When you log a meal, the phone sends it over
  the internet to the backend, which saves it in a database. Next time you open
  the app, the phone asks the backend for your data. Think of the phone as a TV
  remote and the backend as the cable company — the remote shows stuff but the
  real content lives elsewhere.

  ---
  3. The data structures — the heart of your question

  There are two places data is "shaped," and it's important to understand
  they're separate:

  A) Database tables (the permanent shape) — 
  supabase/migrations/0001_phase0_schema.sql

  A table is a spreadsheet. Each row is one record (one meal, one user). Each
  column is one field (calories, protein). You have three main tables:

  profiles — one row per user. Their username, timezone, and their daily goals
  (goal_calories, goal_protein_g, etc.).

  foods — a shared catalog of food items (name, serving size, macros per
  serving).

  meal_logs — one row every time someone logs eating something. This is the
  busiest table. Here's a row's important columns:
  user_id      → which user (links to profiles)
  free_text    → what they typed, e.g. "2 eggs"
  calories, protein_g, carbs_g, fat_g  → the macros
  quantity     → how many servings
  meal_type    → breakfast / lunch / dinner / snack
  eaten_at     → when they ate it

  Now, the genuinely smart parts of this schema that are worth understanding:

  Snapshotting. The macros are copied into the meal_logs row, not just linked to
  the foods table. The code comments explain why: "Snapshotted at insert time
  so a food edit cannot retroactively change logged history." Meaning — if you
  later fix a typo in the "chicken breast" food entry, your meals from last week
  don't silently change. The past is frozen. This is a real-world data-modeling
  decision, and a good one.

  Constraints (the check (...) lines) — rules the database itself enforces and
  refuses to break, no matter what buggy code sends it. Example:
  constraint profiles_goal_calories_min check (goal_calories > 1400)
  This says "a calorie goal must be over 1400." If your app has a bug and tries
  to save 200, the database slams the door. The database is the last line of
  defense — it doesn't trust the app.

  Row Level Security (RLS) — these lines:
  create policy "read own meals" on public.meal_logs
    for select using (user_id = auth.uid());
  This is critical and you should understand it. auth.uid() is "the ID of
  whoever is currently logged in." This policy means: when someone asks the
  database for meal logs, it only returns rows where user_id matches them. So
  User A physically cannot see User B's meals — even though all meals live in
  one shared table. The database enforces privacy per-row. Without RLS, anyone
  could read everyone's data.

  Idempotency — this line:
  create unique index meal_logs_idempotency on public.meal_logs(user_id,
  client_request_id);
  "Idempotent" means "doing it twice has the same effect as doing it once."
  Phones have flaky internet. If you tap "Log meal" and the request is slow, the
  app might send it twice. Each request carries a unique client_request_id. The
  database refuses to insert two rows with the same ID, so you don't get a
  duplicate meal. You'll see the app handle this gracefully in mealLogService.ts
  (the isIdempotencyViolation part — it catches the "duplicate" error and just
  returns the meal that already saved).

  B) TypeScript types (the in-app shape) — src/types/index.ts and 
  src/services/mealLogService.ts

  These describe the same data, but as the app code sees it. An interface is
  just a named shape — a contract saying "an object of this kind has these
  fields." Example:
  export interface MealLog {
    id: string;
    calories: number;
    proteinG: number;   // note: camelCase
    mealType: MealType;
    ...
  }

  ⚠️ Key thing to notice: the database uses protein_g (snake_case) but the app
  uses proteinG (camelCase). These are two different naming conventions, and
  your code has a translation layer. Look at mapMealLog() in mealLogService.ts —
  its entire job is to convert a database row into an app object:
  proteinG: normalizeNumber(row.protein_g),
  This is normal and good practice. The database has its style, the app has its
  style, and one function in the middle translates. (normalizeNumber exists
  because databases sometimes hand back numbers as text like "45.0", so it
  forces them into real numbers.)

  There's also a union type worth knowing:
  export type MealType = 'breakfast' | 'lunch' | 'dinner' | 'snack';
  The | means "or." This says a meal type can only be one of those four exact
  words. Type any other word and TypeScript yells at you. It's a way to make
  invalid states impossible.

  ---
  4. How it all flows together — follow one meal

  Let me trace logging "2 eggs" end to end, because seeing the flow is how it
  clicks. Your files map onto a layered pipeline:

  SCREEN (MealLoggerScreen.tsx)         ← user taps buttons
     ↓ calls
  HOOK (useMealLogger.ts)               ← screen logic helper
     ↓ calls
  SERVICE (mealLogService.ts)           ← talks to the backend
     ↓ uses
  LIB (lib/supabase.ts)                 ← the phone↔backend connection
     ↓ internet ↓
  SUPABASE BACKEND (Postgres + RLS)     ← validates & stores forever

  Each layer has one job. This separation is deliberate — the screen worries
  about looks, the service worries about data, and they don't tangle.

  Definitions for that diagram:

  - Screen — one full page of the app. HomeScreen.tsx, MealLoggerScreen.tsx,
  etc. (in src/screens/).
  - Component — a reusable chunk of UI smaller than a screen, like
  FoodLogItem.tsx (one row in a list). Screens are built out of components.
  - Hook — a reusable bundle of logic that a screen can plug in. They're named
  useSomething by convention. useDailyTotals.ts is a great example: a screen
  says const { meals, totals, goals, isLoading } = useDailyTotals(today) and the
  hook handles all the messy work of fetching the day's meals, adding them up,
  and tracking whether it's still loading. The screen just gets clean results.
  - Service — plain functions that talk to the backend. logMeal(),
  getMealsForDay(), deleteMeal(). No UI here at all — just data in, data out.

  So the actual sequence when you log "2 eggs":
  1. MealLoggerScreen collects what you typed and tapped.
  2. It calls a function from useMealLogger, which calls logMeal(...) in the
  service.
  3. logMeal first validates everything (validateLogMealParams) — is calories a
  real non-negative number? Is mealType one of the four words? This is the first
  line of defense, on the phone.
  4. It sends the data through the supabase client to the backend.
  5. The backend re-checks with its constraints and RLS — the second, untrusting
  line of defense.
  6. It saves a row and sends it back.
  7. The translation layer (mapMealLog) converts the saved row to an app object.
  8. The screen refreshes and you see your meal + updated daily totals.

  Notice validation happens twice (phone and database). That's not redundant
  sloppiness — the phone check gives fast, friendly errors; the database check
  is the unbreakable guarantee, because you can never fully trust data arriving
  from a phone.

  5. State management — how screens share data

  Open question this answers: "if the backend owns the data, what does Zustand
  do?"

  State = data the app is holding right now in memory while it's open (gone when
  you close it). The problem: your Home screen and your Profile screen both
  need to know "who's logged in." How do they share it without passing it
  hand-to-hand through every screen in between?

  That's Zustand (src/store/userStore.ts). It's a single shared box of data
  ("store") any screen can read from or write to directly. Your store holds the
  current user and their goals. In App.tsx, when Supabase confirms a login, it
  calls login(...) to put the user into the store, and the whole app instantly
  knows.

  One detail in your store shows mature thinking. The comments stress that
  XP/points/streaks come only from the backend, never from the app guessing:
  // This is the ONLY way XP/points/streaks change on the client now — the 
  values
  // come from the database (migration 0005's award trigger), never from local 
  guesses.
  Why care? If the phone could award its own points, a user could cheat by
  editing the app, or two devices could disagree. Making the backend the single 
  source of truth keeps everyone honest and consistent. refreshStats()
  re-fetches the real numbers from the server.

  ---
  6. The two pieces of "backend code" you have

  Most of your backend is just the database (tables + rules). But you also have
  two kinds of server-side logic:

  Migrations (supabase/migrations/*.sql) — these are the instructions that build
  your database, written in SQL (the language for talking to databases).
  They're numbered (0001, 0002...) and run in order. Each one is a change to the
  database structure: 0001 creates the tables, 0005_gamification adds the
  XP/streak system. Think of them as version history for your database's shape —
  like git commits, but for table structure. You never edit an old one; you add
  a new numbered one.

  Edge Function (supabase/functions/estimate-meal/) — this is custom code that
  runs on the server, not the phone. ("Edge" just means it runs on servers near
  the user for speed.) Yours estimates the nutrition of free-text food. The flow
  from its own comments:
  Expo app -> estimate-meal -> USDA FoodData Central -> Supabase cache -> app
  You type "2 eggs and toast," it asks USDA (the US government's free
  food-nutrition database) for the numbers, optionally uses an AI model to split
  a multi-item meal into ingredients, and returns an editable estimate. Two
  reasons this lives on the server, not the phone:
  1. Secrets. The USDA/OpenAI API keys live here only ("never logged or returned
  to the client"). If they were in the phone app, anyone could rip them out and
  abuse them.
  2. Caching. It saves USDA answers for 7 days (CACHE_TTL_DAYS = 7) so it's not
  re-asking for the same food constantly — faster and cheaper.

  ---
  7. A few smaller terms you'll keep seeing

  - async / await / Promise — talking to the backend takes time (internet
  round-trip). A Promise is an IOU: "I'll have the answer later." await means
  "pause here until the IOU pays out." async marks a function that's allowed to
  wait. That's why logMeal is async — it waits on the network.
  - useEffect / useState (in App.tsx, hooks) — React building blocks. useState =
  "remember this value and re-draw the screen when it changes" (e.g. loading).
  useEffect = "run this code when the screen first appears or when something
  changes" (e.g. checking if there's an existing login session on startup).
  - Authentication ("auth") — proving who you are (login). Supabase handles
  this. Once logged in, the user has a session (a token proving identity) that
  rides along on every request, which is how auth.uid() knows who's asking.
  - null — means "no value / unknown." Your code is careful that an unknown fat
  value stays null and is never faked as 0 (see the FatSubtypeTotal comments) —
  "unknown" and "zero" are genuinely different facts.

  part that really makes it click:

  SQL is the language you use to talk to the database — to create tables, save
  rows, and ask for data back. But the thing that actually keeps private info

  ---
  The core idea: UI is built from nested "components"

  The whole frontend is built out of components. A component is a reusable
  function that returns a piece of screen. That's it — a function that describes
  "what should appear." You nest small ones inside bigger ones like LEGO
  bricks:

  App
   └─ MainNavigator        (decides which screen shows)
       └─ HomeScreen        (a full page)
           ├─ MacroRing      (the calorie/protein circles)
           ├─ FoodLogItem    (one meal row) ×N
           └─ ChallengeCard  (the challenge box)

  Big components are assembled from smaller ones. HomeScreen doesn't draw a meal
  row itself — it hands that job to FoodLogItem. That's the single most
  important pattern in the whole frontend.

  ---
  JSX: HTML-looking code inside JavaScript

  Look at what FoodLogItem returns (line 50):

  <View style={styles.container}>
    <Text style={styles.name}>{meal.freeText}</Text>
    <Text style={styles.cal}>{formatMacro(calories)} cal</Text>
  </View>

  That XML-looking stuff is JSX — a way to write what the UI looks like inside
  your code. Two things to know:

  - <View> is a box/container (think: a div if you've seen websites — just a
  rectangle to group things and arrange them). <Text> is the only way to show
  words in React Native; you can't just type text loose, it must be wrapped in
  <Text>.
  - The curly braces { } are an escape hatch back into real code.
  {meal.freeText} means "drop the value of the variable meal.freeText right
  here." {formatMacro(calories)} means "run this function and put its result
  here." So JSX is a template with live values plugged in.

  These are React Native's building-block components — View, Text, ScrollView,
  TouchableOpacity — imported at the top: import { View, Text, ... } from 
  'react-native'. They're the raw primitives; everything you built is made of
  these.

  ---
  Props: passing data into a component

  How does FoodLogItem know which meal to show? It's passed in. Look at
  HomeScreen line 134:

  daily.meals.map((meal) => <FoodLogItem key={meal.id} meal={meal} />)

  meal={meal} is a prop ("property") — an input you hand to a component, like an
  argument to a function. Inside FoodLogItem, it's received here (line 9–11,
  42):

  interface FoodLogItemProps {
    meal: MealLog;            // "I require one prop called meal"
  }
  export default function FoodLogItem({ meal }: FoodLogItemProps) { ... }

  So data flows downward: the screen owns the list of meals, and hands each one
  down into a FoodLogItem. This "data flows down via props" is the mental model
  for the entire UI.

  That .map(...) is also worth noting — it's how you turn a list of data into a
  list of UI. "For each meal in the array, produce one FoodLogItem." That's how
  a dynamic number of rows gets rendered. (The key={meal.id} gives each row a
  unique ID so React can track them efficiently — you'll get a warning if you
  forget it.)

  goals

  - Hook = "go get the data and keep it fresh"
  - Component = "given the data, here's what it looks like"

  show-this-if-true : otherwise-this. Reading it top to bottom: if error → error
  message; else if loading → spinner text; else if no meals → empty message;
  else → the actual list. Handling loading/empty/error states like this is the
  mark of a real app vs. a demo.

  ---
  Re-rendering: the magic that makes it feel alive

  Here's the concept that makes React click. When data changes, the component 
  automatically re-runs and the screen updates. You never manually say "go
  change that text." You change the data, and the UI re-draws itself to match.

  Concrete example from your Home screen (lines 33–37):

  useFocusEffect(
    useCallback(() => {
      daily.refresh();          // re-pull today's meals when this screen is 
  focused
    }, [daily.refresh]),
  );

  useFocusEffect runs code whenever the screen comes into view. So: you log a
  meal on the Log tab → tap back to Home → Home re-fetches → daily.meals now has
  the new meal → the component re-runs → the new FoodLogItem appears. You wrote
  zero code to "insert a row." You just refreshed the data and the UI followed.
  That's the whole React model: UI is a function of data.

  (useCallback and useMemo you'll see around — they're performance helpers that
  say "don't rebuild this value every single re-render unless its inputs
  changed." Not essential to grasp yet; just know they're optimizations, not
  core logic.)

  ---
  Styling: how things look — StyleSheet + your theme

  React Native doesn't use CSS files. Styling is JavaScript objects. At the
  bottom of every component you'll see (HomeScreen line 194):

  const styles = StyleSheet.create({
    container: { flex: 1, backgroundColor: Colors.background },
    logButton: {
      backgroundColor: Colors.primary,

  Then you attach a style by name: <View style={styles.container}>. Each style
  is just properties: colors, spacing, sizes. A few key ones:

  - flex: 1 = "grow to fill the available space." The backbone of layout.
  - flexDirection: 'row' = arrange children left-to-right (default is
  top-to-bottom). This is Flexbox, the layout system — you describe rules

  So when Home appears, here's the full picture:

  1. useUserStore and useDailyTotals (hooks) supply the data — user, meals,
  totals, goals.
  2. HomeScreen returns JSX describing the layout: a header, the macro rings, a
  log button, the meals list, etc.

  for any given data, and React keeps the screen in sync as the data changes.
  You're never manually poking at the screen — you're managing data, and the UI
